# SKILL_05 — DOC_FACTORY

## Purpose

Transform requirements into professional BA documents — by first defining scope, confirming with user, then writing from the correct template.

---

## Pipeline

```
Input (chat history / text / file)
    │
    ▼
Step 1: Assess Input Readiness
    │
    ├─ Too vague → Handoff to SKILL_01
    ├─ Missing context → Handoff to SKILL_02
    └─ Sufficient, or input is generated by SKILL_01 or SKILL_02 
            │
            ▼
        Step 2: Define Scope 
            │  (list use cases, edge cases, NFRs → ask user to confirm)
            ▼
        Step 3: Load Template
            │
            ▼
        Step 4: Draft Document
            │
            ▼
        Step 5: Self-Review
            │
            ▼
        Output
```

---

## Step 1: Assess Input Readiness

Check if input has enough substance to write from.
- Input from SKILL_01 / SKILL_02 → skip heavy assessment, go to Step 2.
- Otherwise verify: actors identified? core functionality described? rough scope exists?

| Input State | Action |
|---|---|
| Vague, no clear requirements | → "Recommend dùng **SKILL_01** để phân tích trước." |
| Missing external context | → "Recommend dùng **SKILL_02** để research trước." |
| Clear enough | → Step 2 |

---

## Step 2: Define Scope

Before writing anything, extract and present scope to user for confirmation.

**Output format:**

```
## Scope

### Main Use Cases
| # | Use Case | Ý nghĩa |
|---|---|---|
| UC-01 | [name] | [mô tả ngắn mục đích] |
| UC-02 | ... | ... |

### Edge Cases (nếu có nhắc đến)
- [edge case 1]

### Non-Functional Requirements (nếu có nhắc đến)
- [performance / security / scalability...]

### Out of Scope (nếu rõ ràng)
- [những gì KHÔNG thuộc phạm vi]
```

**After listing, ask user:**
> "Scope trên đúng chưa? Nếu OK, bạn muốn tôi viết loại tài liệu nào? "

**Rule:** If the scope has more than 8–10 complex use cases → suggest splitting into multiple separate docs instead of packing them into one doc.

→ Only proceed to Step 3 when the user confirms.

---

## Step 3: Load Template

User specifies the document type → load the corresponding template:

| Document Type | Template Path | When to Use |
|---|---|---|
| BRD | `template/BRD/BRD_template.md` | Business Requirements Document |
| Use Case Spec | `template/Use Case/Usecase_template.md` | Use Case specification |
| User Story | `template/User Story/User_story_template.md` | User Stories với Acceptance Criteria |
| API Spec | `template/Sequence Diagram/sequence-diagram.md` | Sequence diagram / API documentation |
| ERD | `template/ERD/erd_template.md` | Entity-Relationship documentation |

**Follow template structure exactly — do not make up formats.**

> If the user requests a document type for which there is no template → inform the user and ask them which format they want to use.

## Step 4: Draft Document

### 4.1 Thinking Before Writing

For **every** section of the document, apply this mental checklist:

**Think like a developer:** What questions will they have? Answer preemptively.
**Think like QA:** What will they test? Make it testable.
**Think like a user:** What will frustrate or confuse them? Address it.

### 4.2 Critical Thinking

Always question while writing:
- "Is this requirement complete?"
- "What happens if this fails?"
- "Are there unstated assumptions?"
- "What's the 'why' behind this requirement?"

Proactively identify and document:
- Missing error states
- Undefined user permissions
- Unclear data validation rules
- Absent performance requirements
- Missing integration touchpoints

### 4.3 Edge Case Coverage

For **every** functional requirement or use case, think through:

**Business Edge Cases:**
- Default/initial states — what shows when there's no data?
- Active/inactive states — when should elements be enabled/disabled?
- Empty states — no results, no data
- Boundary conditions — min/max values
- Concurrent operations — two users editing the same item
- Permission changes — user loses access mid-session

**Technical Edge Cases:**
- Network failures (timeout, no connection, slow connection)
- Server errors (500, 503, rate limiting)
- Invalid data (malformed input, injection attempts)
- Browser/device compatibility

**For each edge case, document:**
```
Scenario: [What went wrong]
Detection: [How system identifies]
User Feedback: [What user sees]
System Action: [What system does]
Fallback: [Alternative if primary handling fails]
```

### 4.4 Number Sourcing

Every numbers (KPI, SLA, threshold) must have an annotation sources or their own formula:
```
Avg.scam calls per day ← Formula: (total_call - safe_call) / total_call x 100
Response time ≤ 200ms   ← Source: PRD-stated (section 3.2)
Uptime ≥ 99.9%          ← Source: Industry-standard
Max file size: TBD      ← Source: TBD — confirm with Tech Lead
```

### 4.5 Sensitive Data

If the document describes any of the following data types, add a note to indicate that: `⚠ Sensitive data. Legal check required.`

| Category | Examples |
|---|---|
| Personal identifiers | Full name, national ID(CCCD,CMND), passport, DOB |
| Contact data | Phone, email, address |
| Financial data | Bank account, card number, transaction history |
| Auth data | Password, OTP, token, PIN |
| User-generated content | Chat, private notes, uploaded files |
| Health / biometric | Medical records, face ID, fingerprint |
| Location | GPS, real-time location |

## Step 5: Self-Review

- [ ] Follows template structure exactly
- [ ] Every use case has clear actor, pre/post conditions
- [ ] Every functional req has acceptance criteria
- [ ] Edge cases covered
- [ ] Dependencies identified
- [ ] No ambiguous language
- [ ] Business value clear for each item
- [ ] Numbers have source annotations
- [ ] Open questions and assumptions listed

## Output

- Only output final document + any notes if needed. Do not write introduction or additional explanations.
